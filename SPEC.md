# SPEC.md — 藏经阁知识库规范

> 本文档是藏经阁的执行规范，所有知识条目的创建、修改、删除、评审均须遵守。
> 版本迭代时直接修改本文档，通过 git history 追溯变更。

---

## 1. 定位与目标

藏经阁是 glm-扫地僧 的**私有技术知识库**，不是文档库，不是教程库。

- **面向对象：** glm-扫地僧 自身检索使用，非人类阅读文档
- **核心价值：** 经过验证、去噪、提纯后的结构化知识，替代不可靠的网络搜索
- **使用场景：** 当 Ling 提问某个技术问题时，优先从藏经阁检索已验证的知识，而非每次都去网上搜索

## 2. 信源规范

### 2.1 信源优先级

| 等级 | 类型 | 示例 | 置信度上限 |
|------|------|------|------------|
| **P0** | 内核源码 + 内联文档 | `kernel/Documentation/`, 源码注释 | 5 |
| **P1** | 官方文档/规范 | kernel.org, Vulkan Spec, GPU vendor whitepaper | 5 |
| **P2** | 权威社区 | LKML patch, DRM maintainer 回复, Phoronix | 4 |
| **P3** | 技术博客（仅补充） | LWN, Collabora blog, Daniel Vetter 博客 | 3 |

### 2.2 信源强制规则

- 每条知识条目**必须**附带 `## Sources` 章节
- Sources 中每条引用**必须**标注信源等级（P0/P1/P2/P3）
- P0/P1 信源**必须**提供具体路径（如 `drivers/gpu/drm/drm_atomic.c:drm_atomic_commit()`）
- P2/P3 信源**必须**提供可访问的 URL
- **无信源的知识不入库。** 无法确认的信息标注 `[待证实]`，置信度最高为 2

### 2.3 多源交叉验证

- 同一知识点，优先从 P0/P1 信源获取
- 若 P0 与 P1 冲突，以 P0（源码）为准
- 若 P2/P3 信息与 P0/P1 矛盾，以 P0/P1 为准，并在条目中标注 `[与 XX 来源存在冲突，以源码为准]`
- 仅 P2/P3 信源可获取的信息，置信度不超过 3

## 3. 时效性与版本规范

### 3.1 基准版本

- Linux 内核相关：以**最新 LTS 版本**为基准（当前 6.6）
- 图形 API：以**最新规范版本**为基准（如 Vulkan 1.3, DX12 Ultimate）
- AI 框架：以**最新稳定版**为基准

### 3.2 版本差异标注

同一知识点在不同版本下存在差异时，必须标注：

```
@since v5.15  drm_atomic_helper_commit_planes_on_crtc() 引入
@changed v6.6 参数列表变更，新增 crtc_state 参数
@deprecated v6.8 该函数已废弃，请使用 drm_atomic_helper_commit_planes()
@removed v7.0 该接口已移除
```

### 3.3 过期知识处理

- 定期巡检（每周）检查上游是否有 breaking change
- 发现过期知识：标注 `[已过期]`，创建新版本条目，保留旧条目供版本追溯
- **禁止直接删除旧知识**，通过 git history + 标注实现版本管理

## 4. 目录结构规范

### 4.1 三级树状结构

```
KB-{MODULE}/                    ← 第一级：知识领域
├── _INDEX.md                   ← 模块总入口：概述 + 子模块导航 + 快速定位
├── _GLOSSARY.md                ← 模块术语表
├── _REVIEW-LOG.md              ← 评审日志
│
├── {submodule}/                ← 第二级：子模块
│   ├── _INDEX.md               ← 子模块索引：条目列表 + 关联图
│   ├── {topic}.md              ← 第三级：原子知识条目
│   └── {topic}.md
│
└── {submodule}/
    └── ...
```

### 4.2 文件命名规则

- 索引文件：`_INDEX.md`、`_GLOSSARY.md`、`_REVIEW-LOG.md`（下划线前缀）
- 知识条目：小写英文，短横线分隔，如 `commit-flow.md`、`plane-state.md`
- 禁止中文文件名

### 4.3 文件职责

| 文件 | 职责 | 内容要求 |
|------|------|----------|
| `_INDEX.md` | 导航节点 | 概述 + 子模块列表 + 快速定位表 |
| `_GLOSSARY.md` | 术语锚点 | 术语名 + 一句话定义 + 首次出现位置 |
| `_REVIEW-LOG.md` | 评审追溯 | 每条评审的时间、条目ID、结果、原因 |
| `{topic}.md` | 原子知识条目 | 严格遵循条目模板（见第5节） |

## 5. 知识条目模板

每条知识条目必须严格遵循以下结构：

```markdown
# {短标题}

> {一句话定义，不超过30字}

## Meta
- **ID:** KB-{MODULE}-{submodule}-{NNN}
- **置信度:** {1-5}
- **信源等级:** {P0/P1/P2/P3}
- **验证日期:** {YYYY-MM-DD}
- **适用版本:** {版本范围}
- **关键词:** {逗号分隔}
- **前置知识:** [→ {关联条目}] [→ {关联条目}]
- **关联条目:** [→ {关联条目}] [→ {关联条目}]

## 概述
{2-3句话，说明：是什么、解决什么问题、在整体架构中的位置}

## 架构/调用链
{用文本表示核心调用链或数据流，精简到函数/结构体级别}

## 关键接口

### 函数
- `{function_name}({params})` — {一句话说明}
  - 参数: {参数说明}
  - 返回: {返回值说明}
  - 源码: {具体文件路径}
  - 注意: {关键注意事项}

### 数据结构
- `struct {name}` — {一句话说明}
  - 关键字段:
    - `{field}` — {说明}
  - 源码: {具体头文件路径}

## 实现要点
{源码级别的关键实现细节，不是教程，是"源码里真正做了什么"}

## 版本差异
| 版本 | 变更 |
|------|------|
| v{x.x} | {描述} |

## 陷阱与注意事项
- {常见错误或易忽略的点}

## 使用模式
{2-3个典型的正确使用模式，代码片段级别}

## Sources
- [P0] `{具体路径}` — {具体章节/函数}
- [P1] `{URL}` — {具体段落}
```

### 模板各章节说明

| 章节 | 目的 | 禁止事项 |
|------|------|----------|
| Meta | 机器检索的元数据 | 禁止留空 |
| 概述 | 30秒内理解这个知识点 | 禁止超过3句话 |
| 架构/调用链 | 快速定位代码路径 | 禁止写教程式描述 |
| 关键接口 | 精确的函数签名和结构体 | 禁止省略参数和返回值 |
| 实现要点 | 源码级实现细节 | 禁止复制粘贴大段源码，只写关键逻辑 |
| 版本差异 | 跨版本兼容性速查 | 禁止猜测，必须有依据 |
| 陷阱与注意事项 | 避坑指南 | 禁止空泛描述，必须具体 |
| 使用模式 | 正确用法速查 | 禁止写完整教程，只写关键代码片段 |
| Sources | 信源追溯 | 禁止无路径引用 |

## 6. 索引规范

### 6.1 模块索引 `_INDEX.md`

```markdown
# KB-{MODULE} — {模块名称}

## 概述
{3句话说明这个模块是什么、涵盖范围}

## 子模块导航
| 子模块 | 说明 | 条目数 |
|--------|------|--------|
| [{submodule}/]({submodule}/) | {说明} | {N} |

## 快速定位
- "{用户可能的问题}" → [→ {submodule}/{topic}]
```

### 6.2 子模块索引 `{submodule}/_INDEX.md`

```markdown
# {子模块名称}

## 概述
{2句话}

## 条目列表
| 条目 | 说明 | 关联 |
|------|------|------|
| [{topic}]({topic}.md) | {一句话说明} | → {关联条目} |

## 关联图
{用文本画出条目间的依赖关系}
```

### 6.3 交叉引用规则

- 条目间引用格式：`[→ {topic}]` 或 `[→ {submodule}/{topic}]`
- 引用的目标条目**必须存在**，否则评审不通过
- **禁止复制内容**到其他条目，只通过引用关联
- 前置知识表示"理解本条目需要先了解的内容"
- 关联条目表示"与本条目相关的其他条目"

## 7. 术语规范

### 7.1 术语表 `_GLOSSARY.md`

每个模块维护独立的术语表：

```markdown
# {模块} 术语表

| 术语 | 定义 | 首次出现 |
|------|------|----------|
| CRTC | Cathode Ray Tube Controller，DRM 中表示显示控制器 | KB-LINUX-DRM-crtc-001 |
| Plane | 硬件叠加层，表示一个可独立扫描的图像源 | KB-LINUX-DRM-plane-001 |
```

### 7.2 术语一致性

- 同一概念在所有条目中必须使用同一术语
- 若存在别名，在术语表中标注，条目中使用首选术语
- 评审时检查术语一致性

## 8. 评审规范

### 8.1 评审 Checklist

| # | 检查项 | 规则 | 不通过处理 |
|---|--------|------|------------|
| 1 | 信源可追溯 | Sources 中每条引用有具体路径/URL | 退回补充 |
| 2 | 信源等级合理 | P3 信源置信度 ≤ 3 | 强制降级 |
| 3 | 版本标注 | 必须标注适用版本范围 | 退回补充 |
| 4 | 交叉引用有效 | 所有 `[→ xxx]` 引用的条目必须存在 | 标记断链 |
| 5 | 内容冲突 | 与已有条目是否存在矛盾 | 标记冲突，人工裁决 |
| 6 | 术语一致性 | 术语必须与 `_GLOSSARY.md` 一致 | 统一术语 |
| 7 | 信息密度 | 禁止空话/废话/教程式描述 | 退回重写 |
| 8 | 模板完整 | 所有必填章节不得为空 | 退回补充 |

### 8.2 评审结果

- **PASS** — 条目入库，记录到 `_REVIEW-LOG.md`
- **FAIL** — 条目退回，记录原因，修改后重新评审
- **CONFLICT** — 与已有条目矛盾，标记等待裁决

### 8.3 评审日志格式

```markdown
## {YYYY-MM-DD} | {条目ID} | {PASS/FAIL/CONFLICT}

**评审结果：** {结果}

| 检查项 | 结果 | 备注 |
|--------|------|------|
| 信源可追溯 | ✅/❌ | {说明} |
| 信源等级合理 | ✅/❌ | {说明} |
| 版本标注 | ✅/❌ | {说明} |
| 交叉引用有效 | ✅/❌ | {说明} |
| 内容冲突 | ✅/❌/⚠️ | {说明} |
| 术语一致性 | ✅/❌ | {说明} |
| 信息密度 | ✅/❌ | {说明} |
| 模板完整 | ✅/❌ | {说明} |

**处理：** {入库/退回补充/标记冲突}
```

### 8.4 定期复评

- **频率：** 每周一次
- **触发条件：** 上游（内核/API/框架）有新版本发布
- **范围：** 检查所有条目的版本标注是否仍然有效
- **输出：** 复评摘要报告，包含需要更新的条目列表

## 9. 变更管理规范

### 9.1 新增条目

1. 起草条目（遵循模板）
2. 自动评审（8项 checklist）
3. 评审通过 → commit + push
4. 评审不通过 → 修改 → 重新评审

### 9.2 修改条目

- 任何修改必须说明**修改原因**（上游更新/发现错误/补充信息）
- 修改后重新执行评审
- 在条目 Meta 中更新 `验证日期`

### 9.3 删除条目

- **禁止单方面删除**任何知识条目
- 过期条目标注 `[已过期]`，保留供追溯
- 确认需要删除时，必须记录原因到 `_REVIEW-LOG.md`

### 9.4 Git 提交规范

```
{类型}: {模块}/{子模块} {简述}

类型：
  add     — 新增条目
  update  — 修改条目
  review  — 评审记录
  expire  — 标记过期
  refactor — 结构调整（索引/术语表等）
```

## 10. 渐进式查询规范

### 10.1 查询原则

- **先概述后细节：** 首次查询只返回条目的 `概述` + `架构/调用链`
- **按需深入：** Ling 追问时，再返回对应章节的详细内容
- **引用不复制：** 涉及关联知识时，指向关联条目而非重复内容
- **绝不倾倒：** 禁止一次性返回完整条目内容

### 10.2 查询流程

```
Ling 提问
    │
    ▼
检索 _INDEX.md → 定位子模块 → 定位条目
    │
    ▼
读取条目 → 返回 [概述] + [架构/调用链]
    │
    ▼
Ling 追问具体章节 → 返回该章节内容
    │
    ▼
涉及关联知识 → 读取关联条目的 [概述] → 提示 Ling 是否深入
```
